# LOAS 전체 시나리오 검증 가이드

AMR 순회 관측 시나리오를 송출해 **수집 → 페어링 → 큐 → 판정 → LOAS 적재 → 시각화**
전 구간을 end-to-end 로 검증하는 절차서. 대상 환경은 **Ubuntu 24.04 WSL**.

- 작성 기준 경로: `/home/duckking/echopro`
- 송출기: `SocketDaim/scripts/send_loas_scenario.py`
- 부팅 순서 핵심: **SocketDaim → Dumopro WebApp → decision_agent → migrate_010 → PoolerTran**

---

## 1. 시스템 구성 / 데이터 흐름

```
send_loas_scenario.py ─13310(DUST)/13320(CCTV)→ [ingestion-gw] ─INSERT→ gateway_db(dust_inspection, cctv_frame) + JPEG 파일
                                                      │
                                               [correlator] ─UPDATE(cctv_frame.dust_inspection_id)→ ★트리거 enqueue_cctv_transfer
                                                      │                                                     │
                                                      └────────────────→ gateway_db.cctv_transfer_queue ◄──┘
                                                                                  │
                                          [PoolerTran poller] ─폴링→ 이미지 b64 + 분류 → INSERT decision_record(final_decision='pending')
                                                                                  │                            (decision_db)
                                          [decision_agent] ─판정→ UPDATE final_decision(normal/caution/warning/danger)
                                                                                  │
                                          [egress-gw] ─polling(final_decision<>'pending')→ dust_inspection 조인 → INSERT t_inspection (LOAS MariaDB)

   (병렬 시각화 분기)
   gateway_db(v_loas_stations, v_loas_sensor_sample) ─읽기→ [dumopro-poller] → [dumopro-redis] → [dumopro-api] UI :9105
```

- **수집 게이트**: 이동 중(`waypoint_id`=NULL) DUST 는 저장 안 함 + 그 구간 CCTV 도 미저장. 관측 중에만 저장 → Correlator 가 `received_at` ±2초로 페어링.
- **WebApp 분기**: gateway_db 의 LOAS 뷰만 읽는 독립 소비자(판정/egress 체인과 무관). **Redis 가 여기서 가동**.

## 5. Docker 이미지 준비 (로딩)

자체빌드 6 + 베이스 3(postgres:16, redis:7-alpine, mariadb:11) = 총 9종.

> **개발 PC(이 기기)에서 시나리오 테스트만 할 거면 5-A 만 보면 된다. 5-B 는 폐쇄망

> 알아서 빌드/pull 하므로, 인터넷이 되면 5-A 도 생략하고 바로 6장으로 가도 된다.)

### 5-A. 개발 PC — 직접 빌드 (권장 · 인터넷 가능 시)

```bash

docker compose -f decision_agent/docker-compose.yml                   build
```

### 5-B. 폐쇄망 운영서버 전용 — docker save → load (개발 PC 테스트엔 불필요)

> ⚠️ **이 절은 인터넷이 없는 운영서버 배포 시에만 사용한다. 개발 PC 시나리오 테스트에는
> 거치지 않는다(5-A 로 충분).**

외부망 빌드 PC 에서 추출 후 운영서버로 복사:
```bash
# (외부망) 위 5-A 로 빌드/pull 후 tar 추출
docker save -o images.tar \
  socketdaim-ingestion-gw:latest socketdaim-egress-gw:latest \
  socketdaim-admin-ui:latest socketdaim-cleaner:latest \
  decision_agent-decision-agent:latest poolertran-poolertran:latest \
  dumopro_data_analysis_webapp-dumopro-api:latest \
  dumopro_data_analysis_webapp-dumopro-poller:latest \
  postgres:16 redis:7-alpine mariadb:11

# (운영서버) 복사 후 적재
docker load -i images.tar
docker images   # 11종 확인
```
> 자체빌드 이미지 이름은 `<프로젝트>-<서비스>` 규칙으로 compose 가 기대하는 이름과 일치한다.
> 운영(폐쇄망)에서는 각 레포의 `docker-compose.deploy.yml`(image+pull_policy:never)로 기동한다.

## 8. 검증 (송신 후 ~10초 — Correlator 틱 대기)

```bash
PSQL_GW="docker exec -i sd-postgres psql -U postgres -d gateway_db -c"
PSQL_DC="docker exec -i da-postgres-decision psql -U postgres -d decision_db -c"

# (1) 수집 + 페어링
$PSQL_GW "SELECT (SELECT count(*) FROM dust_inspection) dust,
                 (SELECT count(*) FROM cctv_frame) cctv,
                 (SELECT count(dust_inspection_id) FROM cctv_frame) paired;"
# (2) 이동 구간 미저장 확인 (0 이어야)
$PSQL_GW "SELECT count(*) null_wp FROM dust_inspection WHERE waypoint_id IS NULL;"
# (3) 트리거 → 큐
$PSQL_GW "SELECT count(*) queued FROM cctv_transfer_queue;"
# (4) 목표별 저장 (각 9건)
$PSQL_GW "SELECT target_id, count(*) FROM dust_inspection
          WHERE waypoint_id IS NOT NULL GROUP BY target_id ORDER BY target_id;"
# (5) PoolerTran → decision_record → decision_agent 판정 (pending → 판정)
$PSQL_DC "SELECT final_decision, count(*) FROM decision_record GROUP BY final_decision;"
# (6) egress → LOAS MariaDB t_inspection (이미지 포함)
docker exec -i loas-mariadb mariadb -udaimresearch -p'daimresearch1234!' tfoi_web_db_v1 \
  -e "SELECT count(*) total, count(image_data) imgs FROM t_inspection;"
# (7) 저장 CCTV 파일 확인
find storage/cctv -name '*.jpg' | head -1 | xargs file
```

**기대값**

| 확인 | 기대 |
|---|---|
| (1) dust / cctv / paired | ≈36 / ≈72 / ≈72 |
| (2) null waypoint 행 | 0 |
| (3) queued | >0 |
| (4) target_id별 | 100/200/300/400 각 9 |
| (5) final_decision | pending → 잠시 후 normal 등으로 전환 |
| (6) t_inspection | total>0, imgs>0 |

```bash
cd /home/duckking/echopro/SocketDaim

# (1) 단일 서비스 재생성 + 즉시 적용 (가장 흔함)
docker compose up -d --build egress-gw

# (2) 빌드 / 적용 분리
docker compose build egress-gw
docker compose up -d egress-gw

# (3) 캐시 무시 완전 재빌드 (베이스/의존성 갱신 시)
docker compose build --no-cache egress-gw
docker compose up -d --force-recreate egress-gw

# (4) 여러/전체 서비스
docker compose up -d --build ingestion-gw egress-gw   # 특정 다수
docker compose build                                  # SocketDaim 자체빌드 전체
```

> 변경 종류별:
> - 코드(`*.py`) 변경 → `--build` 필요.
> - compose `environment` 만 변경 → `docker compose up -d <서비스>` (빌드 불필요, 재생성만).

### 13-B. 폐쇄망 운영서버 — 재생성된 이미지 재로딩(save → load)

```bash
# (외부망 빌드 PC) 최신 코드로 빌드 후 tar 추출
cd /home/duckking/echopro/SocketDaim
docker compose build egress-gw
docker save -o egress-gw.tar socketdaim-egress-gw:latest
#   용량 절감:  docker save socketdaim-egress-gw:latest | gzip > egress-gw.tar.gz

# (운영서버) tar 복사 후 적재 — 같은 태그를 덮어쓴다
docker load -i egress-gw.tar
#   gzip 했다면:  gunzip -c egress-gw.tar.gz | docker load
docker images | grep socketdaim-egress-gw            # 적재 확인

# (운영서버) 새 이미지로 컨테이너 교체 (deploy compose: image+pull_policy:never)
cd <socketdaim_root>
docker compose -f docker-compose.deploy.yml up -d --force-recreate egress-gw
```

### 13-C. 적용 확인

```bash
docker image inspect -f '{{.Created}}' socketdaim-egress-gw:latest   # 이미지 생성시각
docker inspect -f '{{.Image}}' sd-egress-gw                          # 컨테이너가 쓰는 이미지 ID
docker logs sd-egress-gw | grep -E 'starting_egress|sql_logging_enabled'
docker exec sd-egress-gw printenv EGW_SQL_LOG_ENABLE
docker image prune -f          # rebuild 반복으로 쌓인 dangling 이미지 정리
```

### 13-D. 요약 표

| 상황 | 명령 |
|---|---|
| 개발 PC, 코드 변경 | `docker compose up -d --build <서비스>` |
| 개발 PC, 완전 재빌드 | `docker compose build --no-cache <서비스> && docker compose up -d --force-recreate <서비스>` |
| 개발 PC, env 만 변경 | `docker compose up -d <서비스>` |
| 폐쇄망 추출 | `docker save -o <svc>.tar <이미지>:latest` |
| 폐쇄망 적재 | `docker load -i <svc>.tar` |
| 폐쇄망 적용 | `docker compose -f docker-compose.deploy.yml up -d --force-recreate <서비스>` |
