# EcoPro 분진 모니터링 파이프라인 — 프로그램 구조 및 동작 (2026-06-25)

폐쇄망(air-gap) 운영을 전제로 한 산업용 AMR(자율주행로봇) **분진(DUST) 모니터링 파이프라인**의 4개 구성요소
(**SocketDaim · PoolerTran · decision_agent · Dumopro_Data_Analysis_WebApp**)에 대한 상세 구조·동작 문서.

## 1. SocketDaim — 수집·상관·LOAS 송출 게이트웨이

LOAS(고객사 시스템)와 내부 분석계를 잇는 **수집(ingestion)** 과 **송출(egress)** 의 양방향 게이트웨이.
`gateway_db`(PostgreSQL)와 CCTV 원본 `storage/` 를 소유한다.

  - `handler/`
    - `loas_dust_handler.py` — XML 디코드→`parse_dust_inspection`→**게이트 판정**→dust_inspection INSERT
    - `loas_cctv_handler.py` — JPG 디스크 기록(**KST 경로/파일명**)→cctv_frame INSERT
    - `waypoint_gate.py` — 모듈 전역 공유상태(`allow_save`, `target_id`)와 수집규칙 `should_collect()`
  - `repository/` — `dust_inspection_repo.py`, `cctv_frame_repo.py` 등
- **`egress_gateway/`**
  - `main.py` — 풀(decision_db, gateway_db)·Outbox·Sink·Poller·Sender 부트스트랩
  - `poller.py` — N초 주기 pending 조회 → Sender
  - `sender.py` — dust 조인 → `_build_row` → Outbox → Sink.write(INSERT) → mark_sent
  - `sink.py` — `MariaDbSink`: LOAS t_inspection **순수 INSERT**(권한 Insert+Select → upsert 불가)
  - `outbox.py` — SQLite 멱등 outbox(at-least-once), 재기동 시 drain
  - `repository/` — `decision_repo.py`(fetch_pending/mark_sent), `gateway_repo.py`(cross-DB dust 25컬럼)
- **`admin_ui/`** — FastAPI(9108): station/station_request CRUD, 디스크 위젯, `[지금 정리]`(NOTIFY cleanup_trigger)
- **`cleaner/`** — 매일 03:00 KST 보존정리 + 디스크 85% 초과 시 비상 purge
- **`libs/gw_proto/`** — LOAS 코덱/전송
  - `codec/loas/` — `dust_framing.py`(12B 헤더), `dust_xml.py`(`DustInspectionPayload` 25필드, EUC-KR fallback), `cctv_framing.py`(9B 헤더)

### 1.3 데이터 모델 (`gateway_db`, `init_db.sql`)

**역할(Role)**

| Role | 권한 | 사용처 |
|---|---|---|
| gw_writer | INSERT dust_inspection/cctv_frame 등, cctv_frame은 (dust_inspection_id,paired_at)만 UPDATE | ingestion |
| gw_reader | 전 테이블 SELECT | PoolerTran, Dumopro, egress |
| gw_admin | station/station_request/waypoint_label CRUD | admin-ui |
| gw_cleaner | SELECT+DELETE (video/sensor_sample/log/dust_inspection/cctv_frame) | cleaner |

**핵심 테이블**

- **`dust_inspection`** (LOAS Tfoi v4a, ~32컬럼): `received_at`(TIMESTAMPTZ), `sensor_datetime`, `dust_value`, `dust_alarm`(0=Fault/1=Maint/2=Alert/3=Normal), 좌표 `waypoint_x/y/z`, 쿼터니언 `rot_x/y/z/w`, 미션 ID들(`ugv_id/location_id/map_id/navigation_id/exec_id/plant_id/target_id/mission_id`), **`waypoint_id`**(NULL·0=주행 / !=NULL·!=0=관측), 자세 `inspection_pan/tilt/lift`, `raw_xml`.
- **`cctv_frame`**: `received_at`, `amr_id`, `resolution`(V1080/V720p/V640p), **`file_path`**(`/data/storage/cctv/{amr}/{YYYY-MM-DD}/{HH}/{target}_{KST타임스탬프_ms}.jpg`), `byte_size`, `dust_inspection_id`(FK, **correlator가 채움**), `paired_at`.
- **`station` / `video` / `sensor_sample` / `ingestion_log` / `station_request`**(이름 PK, UPSERT) — 표준 모드/관리용.
- **`waypoint_label`**: target_id↔라벨(관측 개소 메타).
- **View**: `v_loas_stations`, `v_loas_sensor_sample`(dust_inspection을 sensor_sample 형태로 재구성 → Dumopro), `v_inspection_with_frames`.
- **함수**: `loas_station_id(target_id)→UUID`(결정적 매핑).

### 1.4 런타임 동작

**(A) DUST 수신 (13310)**
1. 12B 헤더+XML body 수신 → `parse_dust_inspection` → `DustInspectionPayload`.
2. **수집 게이트** `waypoint_gate.should_collect(payload)`:
   - 현재 규칙 = **`waypoint_id`(!=NULL,!=0) AND `target_id`(!=NULL,!=0)** 일 때만 수집(True).
   - 하나라도 NULL/0 → 주행/미지정으로 보고 **스킵**(dust 미저장, `allow_save=False`).
   - (예전 좌표/자세 6값 비교 방식은 같은 함수의 `if False:` 블럭에 보존 — 되살리려면 True로.)
3. True면 `gate.target_id` 갱신 후 dust_inspection INSERT.

**(B) CCTV 수신 (13320)**
1. 9B 헤더+JPEG 수신.
2. `gate.allow_save`(직전 DUST 결정) 확인 → False면 그 구간 영상 스킵.
3. True면 경로 생성(**KST 표기**: `astimezone(+9)`로 날짜/시/파일명 산출, DB `received_at`은 UTC=동일 인스턴트) → 원자적 파일 쓰기 → cctv_frame INSERT.

**(C) Correlator** — 10초 주기로 cctv_frame을 `received_at ±2초` 윈도우의 dust_inspection에 매칭(최근접 시간 우선) → `dust_inspection_id`/`paired_at` UPDATE.

**(D) Egress** — 5초 주기:
1. decision_db에서 `final_decision != pending AND sent_at IS NULL` 조회.
2. 각 건: `dust_id`로 gateway_db.dust_inspection 25컬럼 조인 → `_build_row`로 t_inspection 행 생성(`event_id = {normal:0,caution:1,warning:2,danger:3}[final_decision]`, `image_data = decision_record.image_b64`).
3. **조건부 적재**: `event_id`가 `EGW_EVENT_ID_FILTER` 목록 밖이면 INSERT 생략하고 sent 처리(재시도 방지).
4. Outbox 적재 → MariaDB INSERT → 성공 시 Outbox 삭제 + `mark_sent`(at-least-once).

## 2. PoolerTran — 큐 소비 · 분석 REST · 판정 레코드 생성

gateway_db의 **CCTV 전송 큐**를 소비해, **waypoint 단위 배치**로 analysis-receiver에 REST 호출하고,
그 결과를 `decision_db.decision_record`(3채널, `final_decision=pending`)로 적재한다.

### 2.1 Docker 서비스 (`docker-compose.deploy.yml`)

- 이미지 `poolertran-poolertran:latest`, 컨테이너 `poolertran`, health 9109, 네트워크 gw-net.
- **소스 gateway_db**: `PT_GW_DB_*`(user `cctv_forwarder`, migrate_010가 생성).
- **싱크 decision_db**: `PT_DECISION_DB_*`(detector role).
- **REST**: `PT_REST_MODE=batch_paths`, **`PT_REST_URL`**(운영: `http://localhost:8080/` 또는 컨테이너→호스트 `host.docker.internal:8080`), `PT_REST_TIMEOUT_SEC=10`, `PT_API_LOGGING`(응답 로깅 토글), `PT_REST_DEMO`(true=더미응답), `PT_REST_DEMO_VERSION`(1=고정 score, 2=waypoint별 프로필).
- **경로 리맵**: `PT_PATH_REMAP_FROM=/data/storage` → `PT_PATH_REMAP_TO=/home/daim/svc/SocketDaim/storage` (REST payload의 file_path 변환용).
- **폴링/배치**: `PT_POLL_INTERVAL_SEC=5`, `PT_BATCH_SIZE`, `PT_USE_LISTEN`(LISTEN/NOTIFY), `PT_CLEAR_QUEUE_ON_START=true`, `PT_QUEUE_MAX_AGE_SEC=21600`(6h 정리), `PT_MAX_ATTEMPTS=10`(DLQ).
- **볼륨(중요)**: storage를 **호스트 경로 동일 마운트** + 기존 /data/storage 호환:
  ```
  - /home/daim/svc/SocketDaim/storage:/home/daim/svc/SocketDaim/storage:ro
  - /home/daim/svc/SocketDaim/storage:/data/storage:ro
  ```
  → 응답 path1(정적 결과, 호스트 절대경로)을 컨테이너에서 그대로 읽어 image_b64로 인코딩하기 위함.

### 2.2 모듈 구조 (`src/poolertran/`)

`main.py`(풀·poller·health), `config.py`(PTSettings), `poller.py`(메인 루프), `rest_client.py`(BatchPaths/Demo + factory),
`health.py`(9109), `repository/queue_repo.py`(cctv_transfer_queue), `repository/decision_producer.py`(decision_record/threshold/DLQ).

### 2.3 런타임 동작 — REST → INSERT → DELETE (순서 엄수)

1. **waypoint 전이 감지**: 큐의 AMR별 최신 waypoint_id를 직전값과 비교 → **이전 waypoint** 배치 처리(다AMR 독립 추적).
2. **배치 클레임**: `cctv_transfer_queue ⨝ cctv_frame ⨝ dust_inspection`을 `FOR UPDATE OF q SKIP LOCKED`로 최대 `PT_BATCH_SIZE`개 잠금.
3. **고아 제거**: 원본 cctv_frame이 cleaner로 사라진 행은 REST 없이 큐에서 삭제.
4. **① REST(1콜)** `send_batch`: payload `{amr_id, target_id, frames:[{received_time(KST), file_path(리맵)}]}` → 응답 `[{score,path1,path2}(정적), {…}(동적)]`.
5. **② decision_record INSERT**:
   - score 추출 `_extract_dual`(body[0]=정적, body[1]=동적), 임계 분류 `_classify`(value>threshold→abnormal).
   - 3채널 결과: `sensor_analysis_result`(dust_value vs threshold "dust"), `anomaly_detection_result`(정적 vs "static"), `object_detection_result`(동적 vs "dynamic").
   - **이미지**: `_static_p1(body)=body[0].path1`(**정적분석 첫 이미지**) → `_encode_b64` → `image_b64`.
   - 대표행 `_representative_row`(dust_value 최댓값)에서 station_id(`{amr}:wp{wp}`)·observation_timestamp·dust_id.
   - `final_decision=pending`, `result_payload=body`(jsonb). `dust_id` UNIQUE(ON CONFLICT DO NOTHING) → 멱등.
6. **③ 큐 DELETE**: ② 성공 후 같은 트랜잭션에서 삭제.
7. **초과분 폐기(A안)**: 클레임 수 == batch_size이고 정상 처리됐으면 해당 waypoint 잔여 큐를 모두 삭제(과배치 방지). 실패 시엔 보존(재시도).
8. **실패 처리**: attempts 증가, `> PT_MAX_ATTEMPTS`면 `decision_db.transfer_dlq`로 이송. 6시간 초과 stale 행은 주기 sweep 삭제.

> **이미지 경로 주의**: 응답 path1은 receiver의 **호스트 절대경로**이고, `_encode_b64`는 역리맵 없이 그대로 읽는다. 그래서 §2.1의 **호스트 경로 동일 마운트**가 있어야 image_b64가 NULL이 아니라 정적 첫 이미지로 채워진다(이후 egress가 t_inspection.image_data로 전달).

## 3. decision_agent — 최종 판정(3채널 → final_decision)

`decision_db`를 소유하고, PoolerTran이 채운 3채널 결과가 모두 도착한 decision_record에 대해
**진리표(alarm_mapping)** 로 `final_decision`(normal/caution/warning/danger)을 확정한다.

### 3.1 Docker 서비스

| 서비스 | 컨테이너 | 이미지 | 포트 |
|---|---|---|---|
| postgres-decision | `da-postgres-decision` | postgres:16 | 2346:5432 |
| decision-agent | `da-decision-agent` | decision_agent-decision-agent | 9107:9107 (admin UI) |

**agent env**: `DA_DB_USER=decision_agent_role`, `DA_POLL_INTERVAL_SEC=5`, `DA_BATCH_SIZE=100`, `DA_ROLE_REFRESH_SEC=300`, `DA_ADMIN_PORT=9107`.
초기화 SQL: `init_db.sql`(스키마/역할) + `seed_mapping.sql`(role 3·alarm 8·threshold 3행).

### 3.2 데이터 모델 (`decision_db`)

**ENUM**: `model_result`(normal/abnormal), `final_level`(normal/caution/warning/danger), `channel_result`(…+pending), `decision_result`(…+pending).

- **`decision_record`**(관측 1건=1행) 핵심 컬럼:
  - `id`(UUID), `station_id`, `observation_timestamp`, `dust_id`(BIGINT **UNIQUE** = 멱등키, gateway_db.dust_inspection.id 참조).
  - 3채널: `anomaly_detection_result`/`object_detection_result`/`sensor_analysis_result`(+각 `*_at`), 기본 pending.
  - **`final_decision`**(decision_result, 기본 pending) ← **decision_agent 출력**, `decided_at`, `mapping_id`(FK alarm_mapping).
  - `result_payload`(jsonb), `image_b64`(정적 첫 이미지), `sent_at`(egress가 채움), `created_at`.
  - 인덱스: pending 부분인덱스, unsent(`sent_at IS NULL AND final_decision<>pending`), (station_id, observation_timestamp).
- **`alarm_mapping`**(8행, 2×2×2 진리표): (iot_sensor, static, dynamic) → final_decision.
- **`classification_threshold`**(3행): `dust=2.0`, `static=0.5`, `dynamic=0.5` (admin 편집 가능).
- **`role_mapping`**(3행): 탐지 role↔컬럼 동적 매핑.
- **`transfer_dlq`**: PoolerTran 포이즌 메시지(frame_id PK, attempts, last_error, source_row).

**역할**: `anomaly/object/sensor_analysis_detector_role`(PoolerTran 채널 기록), `decision_agent_role`(final_decision 갱신), `egress_role`(decision_record SELECT + sent_at만 UPDATE).

### 3.3 최종 판정 규칙 (alarm_mapping 진리표)

| iot_sensor | static | dynamic | **final_decision** |
|---|---|---|---|
| normal | normal | normal | **normal** |
| normal | normal | abnormal | caution |
| normal | abnormal | normal | caution |
| normal | abnormal | abnormal | caution |
| abnormal | normal | normal | warning |
| abnormal | normal | abnormal | **danger** |
| abnormal | abnormal | normal | **danger** |
| abnormal | abnormal | abnormal | **danger** |

해석: 센서 정상 + 비전만 이상 = caution / 센서 이상 + 비전 무 = warning / **센서 이상 + 비전 중 하나라도 이상 = danger**.

### 3.4 런타임 동작

- **Judge**(`judge.py`): 기동 시 alarm_mapping 8행을 dict 캐시 → `judge(iot,static,dynamic)` O(1) 조회 → (final_decision, mapping_id).
- **RoleResolver**(`role_resolver.py`): role_mapping을 캐시(300초 갱신), role→컬럼 매핑.
- **Poller**(`poller.py`): 5초 주기로 `final_decision=pending AND 3채널 모두 != pending` 행을 observation_timestamp ASC, 최대 100개 조회 → Judge로 판정 → `UPDATE … SET final_decision, decided_at, mapping_id WHERE id=? AND final_decision='pending'`(동시성 안전).
- **Admin UI(9107)**: 상태(pending/stuck), role/alarm/threshold 조회·수정, decision 브라우즈(recent/pending/stuck 탭), 수동 강제판정.

### 4.3 런타임 동작

**(A) Poller**
1. 콜드스타트(`backfill.py`): 최초 1회 전 과거 샘플 로드 → 과거 버킷을 frozen으로, 현재 버킷만 live.
2. 1.5초 주기로 커서(`last_id`) 이후 신규 샘플을 배치(≤500) 조회.
3. 샘플을 시/일/주/월 4버킷에 누적(Redis ZSET `live:raw:...`), 박스통계(Q1/Q3/중앙값/울타리/수염/이상치/극단) 재계산(`live:stats:...`).
4. 버킷 경계 통과 + grace(30초) 경과 → **freeze**: `frozen:{station}:{unit}:{bucket}` 기록 + `frozen:index` 갱신, live 키 삭제.
5. 이벤트 발행: `candle_update`/`candle_frozen`/`station_stalled` → `channel:candle:{station}`.

**(B) SSE (단일 멀티플렉스)**
- 서버: `GET /api/stream` 한 연결로 `channel:candle:*` PSUBSCRIBE → 채널명에서 station 추출해 payload에 `station` 필드 주입(`_inject_station`) → 클라이언트별 asyncio.Queue 팬아웃(15초 heartbeat).
- 클라이언트(`sse.js`): 페이지당 EventSource 1개 → `data.station`으로 디먹스해 station별 구독자에 분배. (이전 station별 다중 EventSource → HTTP/1.1 연결 고갈로 F5 지연 → 단일 스트림으로 개선.)

**(C) UI(9105)**
- **Grid 탭**: 전 station을 일 단위 박스플롯 셀로. 클릭 시 상세 탭.
- **Detail 탭**: 단위(raw/시/일/주/월)·범위·이동평균(MA) 컨트롤 + 회귀(다항 적합/신뢰밴드/잔차).
- **Conflict 모달**: station 재등록(UUID 변경) 감지 시 carry_over/start_fresh 선택.

**(D) 주요 API**: `/api/stream`, `/api/chart/{station}`(frozen+live), `/api/raw/{station}`(DB 최근 N), `/api/stations`(+`/sync`,`/conflicts`), `/api/regression/{station}`, `/api/settings`, `/api/health`.

**(E) Redis 키**: `live:raw|stats:…`, `frozen:…` + `frozen:index:…`, `cursor:{station}`, `stations:list`, `pending_conflict:{station}`, `config:runtime`, 채널 `channel:candle:{station}`·`channel:station-sync`.

## 5. 컴포넌트 간 연동 정리

### 5.1 책임 경계

| 데이터 | 생산 | 소비 |
|---|---|---|
| dust_inspection / cctv_frame (gateway_db) | SocketDaim ingestion | PoolerTran(큐), Dumopro(뷰), egress(조인), correlator |
| cctv_transfer_queue (gateway_db) | (트리거/migrate_010) | PoolerTran |
| storage/cctv/*.jpg (호스트) | SocketDaim ingestion | analysis-receiver(읽기), PoolerTran(결과 path1 읽기) |
| 분석 결과(정적/동적 score+이미지) | analysis-receiver(8080) | PoolerTran |
| decision_record (decision_db) | PoolerTran(3채널, pending) | decision_agent(판정), egress(송출) |
| final_decision (decision_db) | decision_agent | egress |
| t_inspection (LOAS MariaDB) | SocketDaim egress | LOAS |
| 분진 캔들/대시보드 | Dumopro poller | Dumopro api/브라우저 |

### 5.2 핵심 키/식별자

- `dust_id` = gateway_db.dust_inspection.id → decision_record.dust_id(UNIQUE, 멱등) → egress 조인키.
- `target_id` = 관측 개소 식별자(파일명·REST payload·t_inspection·waypoint_label 공통).
- `waypoint_id` = 주행(NULL/0) vs 관측(!=NULL,!=0) 게이트 + PoolerTran 배치 단위.
- `final_decision` → `event_id`(normal0/caution1/warning2/danger3) → t_inspection.event_id (+ EGW_EVENT_ID_FILTER로 선별 적재).
- `image_b64` = **정적분석 첫 이미지(path1)** → t_inspection.image_data.

### 5.3 폐쇄망 배포 개요

각 컴포넌트는 외부망에서 `docker save | gzip` → tar 반입 → 서버 `docker load` → `docker-compose.deploy.yml`(`image:`+`pull_policy:never`)로 기동.
analysis-receiver(GPU)는 별도 5.4GB tar로 반입(`server_scripts/deploy_offline.sh`). 모든 서비스는 `gw-net` 공유.

## 6. 한 줄 요약

> **SocketDaim**(수집·상관·LOAS송출) → **PoolerTran**(waypoint 배치 분석·판정 레코드) → **decision_agent**(3채널 진리표 최종판정) → **SocketDaim egress**(LOAS t_inspection 송출) 의 4단 파이프라인이며, **Dumopro WebApp**은 같은 gateway_db 분진 샘플을 캔들 대시보드로 시각화하는 모니터링 평면이다.
